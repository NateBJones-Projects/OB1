# Feature 1.1d-s3: Task engine — `llm_prompt` task type

**Category:** Add handler to task runner

## Done when

- `llm_prompt` task type handler is implemented in the task runner
- Handler gathers data per `gather_config`, formats as context, sends to LLM with `prompt_template`
- LLM response is delivered via the task's `delivery_channel`
- Morning briefing and weekly review can be registered and executed
- This test works:
  ```
  Create a scheduled task: "morning-briefing", trigger: cron, cron: "0 7 * * 1-5",
  task_type: llm_prompt, prompt_template: "You are a personal productivity assistant...",
  gather_config: {"include_actions": {"status": "open"}, "thought_filters": {"days": 7}},
  delivery: email

  Run morning-briefing now
  → Gathers open actions + recent thoughts, sends to LLM, delivers formatted response
  ```

## What to build

### Handler logic

1. Execute all queries defined in `gather_config`:
   - `thought_filters`: query `thoughts` by type, days, topics
   - `search_queries`: run semantic searches via `match_thoughts`
   - `include_actions`: query `actions` by status, tags, due dates
   - `include_calendar`: query `activities` and `important_dates` for upcoming days
2. Format gathered data as structured context text
3. Send to LLM (via OpenRouter or Anthropic API) with `prompt_template` as system prompt
4. Deliver the response via `delivery_channel`

### LLM integration

Use the LLM API endpoint configured via environment variable (e.g., `LLM_API_URL`, `LLM_API_KEY`). The handler sends a single completion request with the prompt template as system message and gathered data as user message.

### Delivery

For this step, implement at minimum `mcp_response` (return directly) and `email` (via Resend API — `RESEND_API_KEY` env var). Other channels (telegram, slack) can be stubs.

## Dependencies

- 1.1d-s1 (schema + MCP tools)
- 1.1d-s2 (cron triggers) — needed for scheduled execution
- Several weeks of manual daily reviews to calibrate the prompt template
