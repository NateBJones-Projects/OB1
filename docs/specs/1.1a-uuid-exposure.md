# Feature 1.1a: UUID exposure in retrieval tools

**Category:** Patch to `supabase/functions/open-brain-mcp/index.ts`

## Done when

- `search_thoughts` returns UUID for each result
- `list_thoughts` returns UUID for each result
- `capture_thought` returns the new thought's UUID in its confirmation
- `list_actions` and `search_actions` return UUIDs (verify — fix if missing)
- This test works in one conversational turn:
  ```
  Search my thoughts for "Integrity Air"
  → Results include UUIDs

  Create an action from that thought: "Call Integrity Air for ductwork estimate"
  → Claude uses the UUID from search as thought_id in create_action
  ```

## Context

Currently `search_thoughts`, `list_thoughts`, and `capture_thought` don't return the thought's UUID. This makes it impossible to link a thought to an action via `thought_id` without manually looking up the ID in Supabase.

## What to build

All changes are in `supabase/functions/open-brain-mcp/index.ts`:

1. **`list_thoughts`** — add `id` to the select and output:
```typescript
// Change:
.select("content, metadata, created_at")
// To:
.select("id, content, metadata, created_at")

// Update output format:
return `${i + 1}. [${t.id}] [${new Date(t.created_at).toLocaleDateString()}] ...`
```

2. **`search_thoughts`** — add `id` to the type annotation and output:
```typescript
// match_thoughts RPC already returns id — include it in output:
t: { id: string; content: string; metadata: Record<string, unknown>; similarity: number; created_at: string; }
parts.unshift(`ID: ${t.id}`);
```

3. **`capture_thought`** — return UUID after insert:
```typescript
const { data: inserted, error } = await supabase.from("thoughts").insert({...}).select("id").single();
confirmation = `[${inserted.id}] ` + confirmation;
```

4. **`list_actions`** and **`search_actions`** — verify these already return `id`. If not, apply the same pattern.

## Dependencies

None. This is the first feature to build — everything downstream needs it.
