# Feature 3.0-s6: Morning briefing

**Phase:** 3 - Unified app
**Category:** Composite home screen - read-only summary across actions, meals, and household data
**Estimated scope:** Single Claude Code session
**Spec file:** `docs/specs/3.0-s6-morning-briefing.md`
**Depends on:** 3.0-s1 (app shell + auth), 3.0-s2 (shared components), 3.0-s3 (Actions), 3.0-s4 (Household), 3.0-s5 (Meals)

---

## Done when

- [ ] `/` renders the Morning Briefing instead of the placeholder "Here's your day."
- [ ] Greeting adapts to time of day and shows the current user's first name
- [ ] Date is shown under the greeting in a friendly format
- [ ] A 3-stat summary row shows overdue, due today, and open action counts
- [ ] Five sections render in this fixed order: Today's actions, Agent activity, Today's meals, Needs attention, Upcoming maintenance
- [ ] Lee (owner) sees all five sections
- [ ] Liv (member) sees the same page without Agent activity
- [ ] Every card or CTA navigates to the relevant screen
- [ ] Page is read-only: no create/edit/delete UI on the home screen

- [ ] Missing agent infrastructure does not crash the page

## Context for Claude Code

This is session 6 of 8 in the unified app. Sessions S3-S5 already built the source screens. This session composes those existing data sets into the default home screen people open every morning.

This page should feel fast, calm, and scannable on mobile. The user should be able to understand what matters in about 30 seconds, then tap into the deeper screen only when needed.

**Important design rule:** this screen is a read-only aggregator. It should reuse existing auth state, query helpers, and shared UI patterns. Do not add a new table, do not invent home-screen-specific persistence, and do not add edit flows here.

**Use the real query layer in the repo, not guessed APIs.** At the time this spec was written, the relevant files are:
- `apps/brain/src/lib/queries/actions.ts`
- `apps/brain/src/lib/queries/maintenance.ts`
- `apps/brain/src/lib/queries/meal-plans.ts`
- `apps/brain/src/contexts/AuthContext.tsx`

If a tiny new read-only helper is needed, add it beside the existing query file instead of creating duplicate query logic inside components.

## Actual repo interfaces to use

Do not assume the APIs from older planning docs. Match the current codebase:

### Auth

From `AuthContext`, you already have:

```ts
const { user, isOwner, loading } = useAuth()
```

Use the same first-name fallback pattern the current home page already uses:
- `user.user_metadata.full_name` first
- fallback to email prefix

### Actions

From `apps/brain/src/lib/queries/actions.ts`:

```ts
getActions('active')
```

This returns all active actions (`open` + `in_progress`). Compute the briefing subsets client-side:
- overdue
- due today
- open total
- today's action list
- stale items

Do not add a second independent fetch for each stat.

### Maintenance

From `apps/brain/src/lib/queries/maintenance.ts`:

```ts
getTasks()
```

This returns all maintenance tasks. Filter client-side for:
- overdue tasks
- tasks due in the next 14 days

### Meals

From `apps/brain/src/lib/queries/meal-plans.ts`:

```ts
getMonday(date: Date)
formatWeekStart(monday: Date)
getMealPlanWeek(weekStart: string)
```

**Important:** the real repo currently stores `meal_plans` as one row per meal entry, not one row per week blob. The Morning Briefing should follow the actual row-per-meal shape already used by S5.

To build Today's meals:
1. Compute the current week's Monday with `getMonday(new Date())`
2. Convert it with `formatWeekStart(...)`
3. Fetch the week rows with `getMealPlanWeek(weekStart)`
4. Filter the rows to today's `day_of_week`

## Screen layout

Everything renders in one vertical scrollable column inside the authenticated app shell. No tabs and no filters on this page.

Suggested top-to-bottom order:

1. Greeting
2. Stat row
3. Today's actions
4. Agent activity (owner only)
5. Today's meals
6. Needs attention
7. Upcoming maintenance

Use the same page padding and mobile rhythm already established elsewhere in `apps/brain`.

---

## 1. Greeting

Top of page, no card wrapper.

Examples:
- `Good morning, Lee`
- `Good afternoon, Lee`
- `Good evening, Lee`

Time buckets:
- before 12:00 -> morning
- 12:00 through 16:59 -> afternoon
- 17:00 and later -> evening

Below the greeting, show today's date in a format like:

```text
Tuesday, April 1
```

Keep this simple. No weather, quote, summary paragraph, or other decorative content.

---

## 2. BriefingStats

Three compact stat boxes in a single row on mobile-friendly cards:

| Metric | Definition |
|--------|------------|
| Overdue | active actions with `due_date < today` |
| Due today | active actions with `due_date = today` |
| Open | total active actions |

Implementation notes:
- Fetch active actions once via `getActions('active')`
- Compute all three counts from that single result set
- Highlight overdue in a danger color when count > 0
- Highlight due today in a warning color when count > 0
- Open can stay neutral

If actions are still loading, render loading placeholders in the stat boxes rather than hiding the whole row.

## 3. Today's actions

**Priority:** highest

This section surfaces the actions that most need attention right now.

### Data rules

Start from the `getActions('active')` result and derive:
- overdue actions first
- then actions due today
- then `in_progress` actions with no due date

Sort within those buckets in the most sensible order:
- overdue: oldest due date first
- due today: earliest due date first
- in progress without date: oldest updated item first if available, otherwise oldest created item first

Limit the visible list to 5 items.

### Display

Reuse the visual pattern from the Actions screen where practical:
- content/title
- due date indicator
- tags if available
- clear tap target

If there are more than 5 matching items, show a footer CTA like:

```text
View all 8 actions ->
```

### Empty state

If no items match:

```text
Nothing due today. Nice!
```

### Navigation

Tapping an item should navigate to `/actions`.

Nice-to-have only:
- `/actions?id=<action-id>` to auto-open a detail panel if the existing actions page already supports this cleanly

Do not build complicated deep linking here unless it is trivial.

## 4. Agent activity

**Priority:** second

This section is only for Lee / owners:

```tsx
{isOwner && <AgentSection />}
```

### Goal

Show recent automation or agent pipeline activity if such data exists. This is a read-only status slice, not a full feed.

### Supported implementation paths

#### Path A: a usable agent/automation table already exists

If the repo or database already exposes a real table/function for recent runs or agent events, add a small read-only query helper and render the latest 3-5 items.

Example card content:
- source name
- short description
- status badge
- relative timestamp

#### Path B: nothing usable exists yet

Return an empty array from a stub helper and render a graceful placeholder:

```text
Agent pipeline coming soon.
```

This is acceptable for v1 and is likely the first-pass implementation.

### Query helper shape

If no existing helper exists, create:

```ts
// apps/brain/src/lib/queries/agent-events.ts
export type AgentActivityItem = {
  id: string
  title: string
  summary: string
  status: 'new' | 'complete' | 'needs_review' | 'info'
  created_at: string
}

export async function getRecentAgentActivity(limit = 5): Promise<AgentActivityItem[]> {
  return []
}
```

Keep it intentionally small and read-only.

### Navigation

- If the agent feed route already exists, tapping a real activity card should navigate to `/agent-feed`
- If only the placeholder route exists, that is still fine

If there is no activity data, the placeholder itself does not need to be tappable.

## 5. Today's meals

**Priority:** third

Show today's meals from the current week plan.

### Data rules

Use the real meal-plan query interface described above. After fetching the current week:
- filter rows to today's day name (`monday` ... `sunday`)
- sort by meal type in a predictable order:
  - breakfast
  - lunch
  - dinner
  - snack

### Display

Render one compact card per meal with:
- meal type label
- recipe name if one is already hydrated in S5's display layer, otherwise custom meal text
- meal-type accent treatment consistent with the Meals screen

If the stored row only has `recipe_id` and not recipe name, the section may:
- either show the `custom_meal` when present
- or do a lightweight recipe lookup if S5 already has a clean helper for that

Prefer the simplest approach that matches the existing Meals implementation. Do not invent a heavy join layer just for the briefing.

### Empty state

```text
No meals planned today.
```

Optional CTA:

```text
Plan meals ->
```

### Navigation

Tapping a meal card or CTA navigates to `/meals`.

## 6. Needs attention

**Priority:** fourth

This is the "stale items" section, but use a more user-facing label in the UI such as `Needs attention`.

### Data rules

Derive from the same active actions fetch:
- actions with `due_date = null` and `created_at` older than 14 days
- actions with `status = 'in_progress'` and `updated_at` older than 7 days

Limit to 3 items.

### Display

Use standard cards with:
- action content
- a muted stale badge or label
- an age hint such as `Created 3 weeks ago` or `No update for 9 days`

### Empty behavior

If there are no stale items, hide the whole section entirely.

### Navigation

Cards navigate to `/actions`.

---

## 7. Upcoming maintenance

**Priority:** fifth

Show maintenance tasks that are overdue or due soon.

### Data rules

Start from `getTasks()` and filter to:
- overdue tasks
- tasks due within the next 14 days

Sort by `next_due` ascending and limit to 3 items.

### Display

Each card shows:
- task name
- urgency badge:
  - overdue
  - due today
  - due in X days
- last completed date as supporting text if available

### Empty behavior

If there are no relevant maintenance tasks, hide the whole section entirely.

### Navigation

Cards navigate to `/household`.

If S4 already supports a maintenance-specific tab/query param, using that is fine, but plain `/household` is sufficient for v1.

## Section rendering rules

- Render sections in fixed priority order even if some are empty or still loading
- Greeting and stat row should appear first and anchor the page quickly
- Today's actions, Agent activity, and Today's meals should show an explicit empty state
- Needs attention and Upcoming maintenance should disappear entirely when empty
- Agent activity should disappear entirely for members (`!isOwner`)

Recommended section labels:
- `Today's actions`
- `Agent activity`
- `Today's meals`
- `Needs attention`
- `Upcoming maintenance`

Reuse the existing shared `SectionHeader`, `Card`, `Badge`, and loading styles from S2 if they exist. Do not create parallel component systems for the home page.

## Loading behavior

This page should feel fast even though it pulls from multiple sources.

### Required behavior

1. Render shell + greeting immediately
2. Start actions, meals, maintenance, and agent queries in parallel
3. Let each section resolve independently
4. Show lightweight loading placeholders per section while waiting

Do not make the page wait for every query before showing content.

### Acceptable implementation patterns

Either of these is fine:
- one parent component that kicks off all fetches in parallel and passes state down
- independent client components that fetch on mount and render their own loading state

Pick the simpler option that matches the existing app style.

### Shared data note

Because both BriefingStats and two briefing sections depend on active actions, avoid repeated network calls if practical. Prefer:
- one action fetch in a parent component, then pass derived slices down

If that introduces too much complexity, two calls are tolerable for v1, but the preferred implementation is a single shared active-actions fetch.

---

## Navigation behavior

Every tappable element on the briefing should route the user to the place where work actually happens:

- action -> `/actions`
- meals -> `/meals`
- maintenance -> `/household`
- agent activity -> `/agent-feed`

This page should not open edit forms, detail drawers, or confirm dialogs directly unless the rest of the app already has an extremely lightweight deep-link pattern in place.

The Morning Briefing is an index, not a workspace.

---

## Owner vs member behavior

The home screen is almost the same for both users.

### Shared between Lee and Liv

- Greeting
- Stat row
- Today's actions
- Today's meals
- Needs attention
- Upcoming maintenance

### Lee / owner only

- Agent activity

### Data scoping reminder

- Actions are user-scoped, so each person sees their own action counts and action cards
- Meals and maintenance are household-scoped, so both people see the same shared household data

This behavior is correct and should not be "normalized" away.

## Guard rails

- Read-only page only: no mutations, forms, edit buttons, FABs, or confirm dialogs
- Reuse existing query files and auth context
- Match the real `meal-plans.ts` row-per-meal API, not an imagined week-blob API
- Reuse shared components from S2 where practical
- Keep agent querying isolated behind a tiny helper so missing infrastructure fails gracefully
- Prefer a single active-actions fetch that powers stats + action-derived sections
- Do not create home-screen-only tables, RPCs, or persistence
- Keep the UI lightweight and mobile-first
