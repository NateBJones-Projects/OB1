## Context for Claude Code

This is session 2 of 8. Session 1 created the app shell at `apps/brain/` with auth, nav, and placeholder routes. This session builds the reusable component library that every data screen (S3–S7) will compose from.

**This is the highest-leverage session in the entire build.** Every pattern set here gets replicated 5+ times across screens. Invest in getting the APIs clean and the mobile UX right.

**Design principles:**
- Phone-first: 375px is the primary viewport. Desktop is a nice-to-have.
- Flat and clean: white cards, thin borders, generous whitespace.
- Consistent: every list screen uses the same Card. Every edit flow uses the same form pattern.
- Data-agnostic: components receive props, not Supabase queries. Screens own their data fetching.

## Component specifications

### Card

The primary display unit across all screens. Every action, maintenance task, recipe, and agent event renders as a Card.

```tsx
type CardProps = {
  title: string
  subtitle?: string
  badges?: BadgeProps[]
  rightContent?: React.ReactNode    // e.g., a due date badge or status indicator
  onTap?: () => void                // opens DetailPanel
  className?: string
}
```

**Behavior:**
- White background, 0.5px border, rounded corners (12px)
- Padding: 12px 16px
- Title: 14px, font-weight 500
- Subtitle: 12px, muted text color
- Badges render below subtitle in a horizontal row
- `rightContent` floats to the right of the title row (use flexbox)
- If `onTap` is provided, entire card is tappable (cursor pointer, subtle hover/active state)
- Minimum height: 48px (touch target)

### CardList

Renders a vertical list of Cards with consistent spacing.

```tsx
type CardListProps = {
  items: CardProps[]
  emptyMessage?: string             // shown when items is empty
  loading?: boolean
}
```

**Behavior:**
- 8px gap between cards
- When `loading` is true, show LoadingSpinner
- When `items` is empty, show EmptyState with `emptyMessage`
- No infinite scroll needed for v1 — just render all items

### DetailPanel

Slide-up panel that covers ~85% of the screen height. Used when tapping a Card to see full details.

```tsx
type DetailPanelProps = {
  isOpen: boolean
  onClose: () => void
  title: string
  children: React.ReactNode          // the detail content
  onEdit?: () => void                // shows edit button in header
  onDelete?: () => void              // shows delete button (owner only)
}
```

**Behavior:**
- Slides up from the bottom of the screen (CSS transform + transition)
- Background overlay dims the content behind it
- Header row: back/close button (left), title (center), edit button (right, if `onEdit` provided)
- Delete button at the bottom of the panel content, red text, only renders if `onDelete` is provided
- Close on overlay tap or close button
- Smooth open/close animation (~200ms ease-out)
- Content area scrolls if it overflows
- `onDelete` should only be passed by screens when `isOwner` is true (from AuthContext). The component itself doesn't check role — the screen decides what to pass.

### EditForm

Generic form wrapper that renders fields from a config array. Used for creating and editing records across all screens.

```tsx
type FieldConfig = {
  name: string                       // field key in the data object
  label: string
  type: 'text' | 'textarea' | 'date' | 'select' | 'number'
  required?: boolean
  placeholder?: string
  options?: { label: string; value: string }[]  // for select type
}

type EditFormProps = {
  fields: FieldConfig[]
  initialValues?: Record<string, any>  // populated for edit, empty for create
  onSave: (values: Record<string, any>) => Promise<void>
  onCancel: () => void
  saveLabel?: string                   // default "Save"
  loading?: boolean
}
```

**Behavior:**
- Renders inside DetailPanel (panel opens with the form as children)
- Each field renders with label above, input below
- Text inputs: full width, 44px height (prevents iOS zoom), 16px font
- Textarea: full width, 120px minimum height, resizable
- Select: native `<select>` element (best mobile UX)
- Date: native `<input type="date">` (uses device date picker)
- Required fields show a subtle indicator (asterisk or red border on empty submit)
- Save button: full width, prominent, bottom of form
- Cancel button: text-only, above or beside save
- `onSave` is async — button shows loading state while saving
- Basic validation: required field check before calling `onSave`
- Does NOT close the panel — the screen's `onSave` handler closes it after success

### FAB (Floating Action Button)

"+" button for creating new items.

```tsx
type FABProps = {
  onTap: () => void
  label?: string                     // screen reader label, default "Add new"
}
```

**Behavior:**
- Fixed position, bottom-right of the screen (16px from edges)
- 56px circle, filled with the app's primary accent color
- White "+" icon centered
- Subtle scale animation on tap (0.95)
- `z-index` above content but below DetailPanel overlay
- Tapping opens the EditForm in a DetailPanel (the screen handles this wiring)

### Badge

Small colored pill for status indicators and tags.

```tsx
type BadgeProps = {
  label: string
  variant: 'red' | 'amber' | 'green' | 'blue' | 'purple' | 'gray'
}
```

**Behavior:**
- Small pill: font 11px, padding 2px 8px, rounded full
- Each variant has a light background + dark text from the same color family:
  - `red`: light red bg, dark red text (for overdue, errors)
  - `amber`: light amber bg, dark amber text (for due today, warnings)
  - `green`: light green bg, dark green text (for complete, success)
  - `blue`: light blue bg, dark blue text (for info, upcoming)
  - `purple`: light purple bg, dark purple text (for agent activity)
  - `gray`: light gray bg, dark gray text (for neutral/default)
- Works in both light and dark mode (use Tailwind dark: variants)

### FilterTabs

Horizontal tab row for filtering lists (e.g., Active / Upcoming / Done).

```tsx
type FilterTabsProps = {
  tabs: { label: string; value: string }[]
  activeTab: string
  onChange: (value: string) => void
}
```

**Behavior:**
- Full-width row, evenly distributed tabs
- Active tab: accent color text + bottom border (2px)
- Inactive tabs: muted text, no border
- Tappable with 44px minimum height
- Horizontally scrollable if more than 4 tabs (rare but safe)

### SectionHeader

Muted label used to group content within a screen.

```tsx
type SectionHeaderProps = {
  label: string
}
```

**Behavior:**
- Uppercase, 11px, muted text color, letter-spacing 0.5px
- Margin: 16px top, 8px bottom
- Used in Morning Briefing to label sections ("Today's actions", "Upcoming maintenance", etc.)

### StatRow

Row of metric boxes for summary numbers.

```tsx
type StatProps = {
  value: number | string
  label: string
}

type StatRowProps = {
  stats: StatProps[]                  // 2–4 items
}
```

**Behavior:**
- Horizontal flexbox, equal-width boxes, 8px gap
- Each box: secondary background, 8px rounded corners, 8px padding
- Value: 20px, font-weight 500, centered
- Label: 11px, muted, centered, below value
- Used at the top of Morning Briefing and potentially Actions screen

### ConfirmDialog

Modal for confirming destructive actions (delete).

```tsx
type ConfirmDialogProps = {
  isOpen: boolean
  title: string                      // e.g., "Delete this recipe?"
  message: string                    // e.g., "This can't be undone."
  confirmLabel?: string              // default "Delete"
  onConfirm: () => Promise<void>
  onCancel: () => void
  destructive?: boolean              // default true, makes confirm button red
}
```

**Behavior:**
- Centered modal with overlay backdrop
- Title, message, two buttons (cancel + confirm)
- Confirm button is red when `destructive` is true
- `onConfirm` is async — button shows loading state
- Close on overlay tap or cancel button

### EmptyState

Friendly message when a list has no items.

```tsx
type EmptyStateProps = {
  message: string                    // e.g., "No actions due today"
  actionLabel?: string               // e.g., "Create one"
  onAction?: () => void
}
```

**Behavior:**
- Centered in the available space
- Muted text, 14px
- Optional action button below the message (text-only, accent colored)

### LoadingSpinner

Simple centered loading indicator.

```tsx
type LoadingSpinnerProps = {
  size?: 'sm' | 'md'                // sm = 20px, md = 32px
}
```

**Behavior:**
- CSS-only spinner (animated border), no external library
- Centered in its parent container
- Muted color

## Implementation notes

**No data fetching in any component.** Every component receives data via props. Screens (S3–S7) own the Supabase queries and pass results down. This keeps components pure and testable.

**Tailwind only.** No component libraries (shadcn, Radix, Headless UI). These components are simple enough that Tailwind classes are cleaner than adding dependencies. The fewer dependencies, the less Claude Code has to manage.

**Dark mode support.** Use Tailwind's `dark:` variants for all color choices. The app should respect the user's system preference. Test the demo page in both modes.

**Animations.** Keep them minimal and CSS-only:
- DetailPanel: `transform: translateY(100%) → translateY(0)` with `transition: transform 200ms ease-out`
- FAB tap: `transform: scale(0.95)` on active
- ConfirmDialog: fade-in overlay + scale-in modal

**No `position: fixed` on DetailPanel.** Use `position: absolute` within a full-screen container instead. Fixed positioning causes issues with mobile viewport height and iOS safe areas. The DetailPanel's parent should be a portal-like wrapper that covers the screen.

**Touch handling.** Card `onTap` should use `onClick` (not touch events). React handles touch-to-click automatically. Don't add custom touch handling unless needed.

## Test prompts

After building, verify these scenarios:

1. **Demo page renders:** Navigate to `/demo` → should see all components with sample data
2. **Card tap → detail:** Tap a card → DetailPanel slides up smoothly, shows content
3. **Detail → edit:** Tap edit button in DetailPanel → form renders with fields
4. **Form validation:** Submit with required field empty → shows error, does not call onSave
5. **Confirm dialog:** Tap delete → ConfirmDialog appears, tap cancel → closes, tap confirm → calls handler
6. **FAB:** "+" button visible, tappable, positioned correctly in bottom-right
7. **Filter tabs:** Tap between tabs → active state moves correctly
8. **Empty state:** CardList with no items shows EmptyState message
9. **Badge variants:** All 6 color variants visible and readable
10. **Dark mode:** Toggle system dark mode → all components remain readable and styled
11. **Mobile viewport:** At 375px width, nothing overflows, all tap targets are ≥44px
