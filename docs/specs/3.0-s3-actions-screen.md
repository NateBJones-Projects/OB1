## Actions table schema

The actions table already exists. Do not modify it. Reference only:

```sql
-- Inferred from MCP tool signatures — verify against schemas/actions/ in the repo
actions (
  id              uuid primary key,
  content         text not null,           -- what needs to be done
  status          text default 'open',     -- open | in_progress | done | cancelled
  due_date        date,                    -- nullable, YYYY-MM-DD
  tags            text[],                  -- array of strings
  recurrence      text,                    -- null | daily | weekly | monthly
  blocked_by      text,                    -- free text describing blocker
  unblocks        text,                    -- free text describing what this unblocks
  thought_id      uuid,                    -- linked source thought (nullable)
  completion_note text,                    -- required when completing
  user_id         uuid references auth.users(id),
  created_at      timestamptz default now(),
  completed_at    timestamptz              -- set when status → done
)
```

**Important:** Claude Code should verify this against the actual schema at `schemas/actions/` in the repo. The above is inferred from MCP tool signatures and may be missing columns (e.g., `next_due`, `step_log` from Features 1.1b and 1.1e). Query the actual table schema before building.

## Screen layout (top to bottom)

### Header area
- Page title: "Actions"
- Search bar below title (collapsible — tap icon to expand, or always visible if space allows)
- StatRow: 3 metrics — "Overdue" (count, red), "Due today" (count, amber), "Open" (total count)

### Filter tabs
- **Active** (default): status = `open` or `in_progress`
- **Done**: status = `done`
- **Cancelled**: status = `cancelled`

### Tag filter (Active tab only)
- Below FilterTabs, show a horizontally scrollable row of tag pills
- Derived from the distinct tags across the user's active actions
- Tapping a tag filters the list to only actions with that tag
- "All" pill at the left to clear the filter
- Only visible on the Active tab (Done and Cancelled show a flat list)

### Action list (Active tab)

Grouped by due date urgency using SectionHeaders:

1. **Overdue** — `due_date < today` and status is open/in_progress
2. **Today** — `due_date = today`
3. **Upcoming** — `due_date > today`, sorted by due_date ascending
4. **No date** — `due_date is null`

Each group only renders if it has items. Empty groups are hidden.

### Action list (Done / Cancelled tabs)

Flat list, sorted by `completed_at` or `created_at` descending (most recent first). No grouping needed.

### FAB
- Bottom-right "+" button
- Tapping opens DetailPanel with ActionForm in create mode

## Component details

### ActionCard

Composes the shared Card component with action-specific content.

```tsx
// Maps an action record to Card props
function ActionCard({ action, onTap }) {
  return (
    <Card
      title={action.content}
      subtitle={buildSubtitle(action)}       // e.g., "Due Apr 5 · Home"
      badges={buildBadges(action)}
      rightContent={buildRightContent(action)}
      onTap={() => onTap(action)}
    />
  )
}
```

**Badge logic:**
- Overdue: `{ label: "Overdue Xd", variant: "red" }`
- Due today: `{ label: "Today", variant: "amber" }`
- In progress: `{ label: "In progress", variant: "blue" }`
- Recurring: `{ label: "↻ weekly", variant: "gray" }` (showing the recurrence type)
- Blocked: `{ label: "Blocked", variant: "gray" }` (if `blocked_by` is set)
- Tags render as gray badges: `{ label: "home", variant: "gray" }`

**Subtitle:** Show due date (formatted nicely, e.g., "Apr 5") and first tag if present, separated by " · ". If no due date and no tags, no subtitle.

### ActionDetail

Content rendered inside the DetailPanel when an action card is tapped.

**Sections (top to bottom):**
1. **Content** — the action text, full width, 16px
2. **Status** — badge showing current status
3. **Due date** — formatted date, or "No due date" in muted text
4. **Recurrence** — if set: "Repeats weekly" / "Repeats monthly" / "Repeats daily". If not set, omit section.
5. **Tags** — horizontal row of tag badges, or "No tags" muted
6. **Blocked by** — if set, show the blocker text with a subtle warning style. If not, omit.
7. **Unblocks** — if set, show what this unblocks. If not, omit.
8. **Linked thought** — if `thought_id` is set, show "Linked to thought" (not tappable in v1). If not, omit.
9. **Created** — formatted date/time
10. **Completed** — if done, show completion note and completed_at. If not done, omit.

**Header buttons:**
- Edit button (always) → opens ActionForm in edit mode
- Complete button (if status is open or in_progress) → opens CompleteDialog
- Delete button (owner only, at bottom of panel) → opens ConfirmDialog

### ActionForm

Configures EditForm with action-specific fields.

**Create mode fields:**
- Content (textarea, required, placeholder: "What needs to be done?")
- Due date (date, optional)
- Recurrence (select: None / Daily / Weekly / Monthly, default None)
- Tags (text, optional, comma-separated — parse to array on save)
- Blocked by (text, optional, placeholder: "What's blocking this?")
- Unblocks (text, optional, placeholder: "What does completing this unblock?")

**Edit mode fields:** Same as create, prepopulated with current values. Plus:
- Status (select: Open / In progress / Done / Cancelled)

**Save handler:**
- Create: `supabase.from('actions').insert({ ... })`
- Edit: `supabase.from('actions').update({ ... }).eq('id', action.id)`
- On success: close panel, refresh the action list
- On error: show error message in the form, don't close

**Tag input:** Accept comma-separated text, trim whitespace, store as `text[]`. Display existing tags as comma-separated string in the input when editing.

### CompleteDialog

A specialized modal for completing an action. Not a simple confirm — it requires a completion note.

```tsx
type CompleteDialogProps = {
  isOpen: boolean
  actionContent: string              // shown as context: "Complete: [action content]"
  onComplete: (note: string) => Promise<void>
  onCancel: () => void
}
```

**Layout:**
- Title: "Complete action"
- Context: the action's content text (read-only, muted)
- Textarea: completion note (required, placeholder: "What was done?")
- Cancel + Complete buttons
- Complete button disabled until note has content

**On complete:** Calls `supabase.from('actions').update({ status: 'done', completion_note: note, completed_at: new Date() }).eq('id', action.id)`. If the action has `recurrence`, also trigger the recurrence logic (see Recurring Actions below).

### ActionSearch

Search bar that queries actions by content text.

```tsx
type ActionSearchProps = {
  onSelect: (action: Action) => void  // opens ActionDetail
}
```

**Behavior:**
- Text input with search icon
- Debounced search (300ms) — queries `supabase.from('actions').select().ilike('content', '%query%')` or uses Supabase full-text search if available
- Results render as a simple list of ActionCards below the search input
- Tapping a result opens ActionDetail
- Empty query hides results and shows the normal list view
- Search across all statuses (not filtered by current tab)

## Recurring actions

Feature 1.1b added `recurrence` (daily/weekly/monthly) and `calculateNextDue` logic. The dashboard needs to handle this correctly:

**Display:**
- Recurring actions show a gray badge with the recurrence type: "↻ daily", "↻ weekly", "↻ monthly"
- The due date shown is the *next* due date (from `next_due` or `due_date` column — verify against schema)

**Completion behavior:**
- When a recurring action is completed, a new instance should be created with the next due date
- Check if this is handled server-side (via database trigger or MCP tool logic). If so, the frontend just calls the complete endpoint and refreshes.
- If not handled server-side, the frontend should: (1) complete the current action, (2) create a new action with the same content, tags, and recurrence, and a due_date calculated from the recurrence schedule.
- **Important:** Claude Code should check the `complete_action` MCP tool and the actions schema to determine if recurrence is handled automatically. Don't duplicate server-side logic.

**Creating recurring actions:**
- The form includes a Recurrence select field (None / Daily / Weekly / Monthly)
- When recurrence is set, due_date becomes required (you need a starting date for the schedule)
- Enforce this in form validation: if recurrence !== None, due_date must be set

## Data fetching pattern

Establish this pattern here — S4 and S5 will follow it:

```tsx
// lib/queries/actions.ts

import { createClient } from '@/lib/supabase/client'

export async function getActions(status: string = 'open') {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('actions')
    .select('*')
    .in('status', status === 'active' ? ['open', 'in_progress'] : [status])
    .order('due_date', { ascending: true, nullsLast: true })

  if (error) throw error
  return data
}

export async function createAction(action: Partial<Action>) {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('actions')
    .insert(action)
    .select()
    .single()

  if (error) throw error
  return data
}

// ... updateAction, deleteAction, completeAction, searchActions
```

**Key decisions:**
- All queries go through `lib/queries/actions.ts` — the page component calls these functions, not raw Supabase calls
- Use the browser client (not server client) since this is a client-rendered interactive screen
- Error handling: queries throw, page component catches and shows error state
- After any mutation (create/update/delete/complete), refresh the list by re-fetching

## Delete behavior

- Owner only: the screen checks `isOwner` from AuthContext before passing `onDelete` to DetailPanel
- Tapping delete opens ConfirmDialog: "Delete this action? This can't be undone."
- On confirm: `supabase.from('actions').delete().eq('id', action.id)`
- On success: close DetailPanel, refresh list
- Members never see the delete button — it's not rendered, not just disabled

## Edge cases to handle

1. **No actions at all:** EmptyState: "No actions yet. Tap + to create one."
2. **No actions in a tab:** EmptyState varies: "No completed actions" / "No cancelled actions"
3. **No actions matching tag filter:** EmptyState: "No actions tagged [tag]"
4. **Overdue action with no due date:** Not possible — only actions with due_date < today are overdue
5. **Action with very long content:** Truncate on the card (2 lines max), show full content in DetailPanel
6. **Tag with spaces:** Trim but allow spaces within a tag (e.g., "home repair" is one tag)
7. **Completing a recurring action while offline:** Don't worry about offline for v1
