## Schema reference

```sql
-- Already exists, do not create
create table households (
  id uuid primary key default gen_random_uuid(),
  name text not null default 'Home',
  created_at timestamptz default now()
);

create table household_members (
  household_id uuid references households(id),
  user_id uuid references auth.users(id),
  role text not null default 'member' check (role in ('owner', 'member')),
  primary key (household_id, user_id)
);
```

## Implementation details

### Supabase client setup

Use `@supabase/ssr` for Next.js App Router. Two clients:

1. **Browser client** (`client.ts`) — used in client components for auth state, realtime, mutations
2. **Server client** (`server.ts`) — used in server components and route handlers for secure data fetching

```ts
// lib/supabase/client.ts
import { createBrowserClient } from '@supabase/ssr'

export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}
```

### AuthContext

Client component context that:
1. Calls `supabase.auth.getSession()` on mount
2. Subscribes to `onAuthStateChange` for login/logout reactivity
3. On authenticated session, queries `household_members` to get `role` and `household_id`
4. Exposes: `{ user, role, householdId, isOwner, loading, signOut }`

```ts
// Type shape
type AuthState = {
  user: User | null
  role: 'owner' | 'member' | null
  householdId: string | null
  isOwner: boolean
  loading: boolean
  signOut: () => Promise<void>
}
```

The `isOwner` convenience boolean is what the nav and future delete buttons check.

### Login page

Simple email + password form. No registration flow needed — users are pre-created in Supabase.

- Email input, password input, submit button
- Error message display for bad credentials
- On success, redirect to `/`
- If already authenticated, redirect to `/`
- Centered on screen, clean and simple, mobile-friendly

### Nav (hamburger drawer)

- Hamburger icon (three lines) in top-left of the app header
- Tapping opens a slide-in drawer from the left
- Tapping outside the drawer or a nav item closes it
- Overlay dims the background when drawer is open

**Nav items (all users):**
- Morning briefing → `/` (with a home icon or label)
- Actions → `/actions`
- Household → `/household`
- Meals → `/meals`

**Nav items (owner only — hidden entirely for members):**
- Divider line
- Section label: "Personal" (muted, small text)
- Agent feed → `/agent-feed`
- Thoughts → `/thoughts`

**Bottom of drawer:**
- Settings (placeholder for now)
- Sign out button

**Active state:** Current route is highlighted in the nav.

### Home page (`/`)

- Greeting: "Good morning, [first name]" / "Good afternoon" / "Good evening" based on time of day
- Below greeting: "Here's your day." (static text for now)
- This page will become the Morning Briefing in S6, but for now it's just the greeting
- Shows today's date formatted nicely (e.g., "Tuesday, April 1")

### Placeholder routes

Each placeholder route (`/actions`, `/household`, `/meals`, `/agent-feed`, `/thoughts`) renders:
- The page title (e.g., "Actions")
- A simple "Coming soon" message
- These exist so the nav links work and so S3–S7 can replace the content without creating new routes

### Auth protection

- All routes except `/login` require authentication
- Use Next.js middleware OR a client-side `AuthGuard` wrapper
- Unauthenticated users are redirected to `/login`
- The `/agent-feed` and `/thoughts` routes should also check `isOwner` — if a member navigates there directly (e.g., via URL), redirect to `/`

## Mobile-first design notes

- Minimum tap target: 44px × 44px for all interactive elements
- Nav header height: ~56px (standard mobile app bar)
- Drawer width: ~280px (standard mobile drawer)
- Body content padding: 16px horizontal
- Use `dvh` (dynamic viewport height) where available for full-height layout
- No horizontal scroll anywhere
- Font sizes: 16px minimum for body text (prevents iOS zoom on input focus)
