# Feature 1.1b: Recurring actions

**Category:** Schema migration + MCP tool updates in `supabase/functions/open-brain-mcp/index.ts`

## Done when

- `recurrence` and `recurrence_source_id` columns exist on `actions` table
- `create_action` accepts optional `recurrence` param (`daily`, `weekly`, `monthly`)
- `complete_action` on a recurring action auto-spawns the next instance with correct due date
- `list_actions` shows `(recurring: weekly)` badge and supports `recurring_only` filter
- Due dates anchor to the schedule, not completion date (see key decisions below)
- These tests pass:
  ```
  Create a recurring action: Weekly review. Recurrence: weekly. Due: Sunday. Tag: productivity.

  Complete action [id]: Completed weekly review — surfaced 3 stale loops.
  → Confirms completion AND shows next instance created with due date = next Sunday.

  List my open actions.
  → Shows the newly spawned weekly review with next Sunday's due date.
  ```

## Context

Repeating tasks (weekly reviews, monthly maintenance checks) need to automatically generate new instances when completed, without piling up uncompleted copies.

## What to build

### Schema migration (`schemas/actions/002_recurring.sql`)

```sql
alter table actions add column recurrence text check (recurrence in ('daily', 'weekly', 'monthly'));
alter table actions add column recurrence_source_id uuid references actions(id) on delete set null;
create index idx_actions_recurrence on actions(recurrence) where recurrence is not null;
```

- `recurrence`: `null` = one-time. `'daily'`/`'weekly'`/`'monthly'` = repeating.
- `recurrence_source_id`: new instance points back to the completed one, creating a chain.

### MCP tool changes in `open-brain-mcp/index.ts`

1. **`create_action`** — add optional `recurrence` param.

2. **`complete_action`** — after completing, if recurring, spawn next instance:
```typescript
if (completedAction.recurrence) {
  const nextDue = calculateNextDue(completedAction.due_date, completedAction.recurrence);
  await supabase.from("actions").insert({
    content: completedAction.content,
    status: 'open',
    due_date: nextDue,
    recurrence: completedAction.recurrence,
    recurrence_source_id: completedAction.id,
    tags: completedAction.tags,
  });
}
```

3. **Due date calculation:**
```typescript
function calculateNextDue(currentDue: string | null, recurrence: string): string {
  const base = currentDue ? new Date(currentDue) : new Date();
  switch (recurrence) {
    case 'daily': base.setDate(base.getDate() + 1); break;
    case 'weekly': base.setDate(base.getDate() + 7); break;
    case 'monthly': base.setMonth(base.getMonth() + 1); break;
  }
  return base.toISOString().split('T')[0];
}
```

4. **`list_actions`** — add optional `recurring_only` boolean filter. Show `(recurring: weekly)` badge in output.

### Key decisions

- New instances spawn on **completion**, not on a cron. No guilt-pile-up.
- Due dates anchor to the **schedule**, not completion date: "Weekly review" due Sunday March 29, completed Friday March 27 → next due Sunday April 5.
- If `due_date` is null on source action, fall back to calculating from today.

- Feature 1.1a (UUID exposure) should be done first so spawned actions can be referenced.
