# Feature 1.1d-s1: Task engine — schema + manual trigger

**Category:** `schemas/scheduled-tasks` + `supabase/functions/scheduled-tasks-mcp/index.ts`

## Done when

- `scheduled_tasks` and `task_run_log` tables exist (schema below)
- MCP tools exist: `create_scheduled_task`, `update_scheduled_task`, `list_scheduled_tasks`, `run_task_now`, `task_run_history`
- `run_task_now` executes a task's gather phase and logs to `task_run_log`
- Only `manual` trigger type works in this step (cron comes in 1.1d-s2)
- Task type handlers are stubs that log "not implemented" — real handlers come in later steps
- `README.md` + `metadata.json` exist in `schemas/scheduled-tasks/`
- These tests work:
  ```
  Create a scheduled task: "test-task", trigger: manual, task_type: llm_prompt, delivery: email
  → Task registered

  List my scheduled tasks
  → Shows test-task with status

  Run test-task now
  → Executes gather phase, logs run to task_run_log, returns stub output

  Show task run history
  → Shows the run with timing and status
  ```

## Context

A general-purpose engine for "Claude takes action based on a trigger." Morning briefings, alerting, stale loop scans, deck prep — all the same shape:

```
Trigger → Data gathering → Execution → Delivery
```

One task runner that executes registered task definitions, instead of standalone Edge Functions per use case.

## What to build

### Schema (`schemas/scheduled-tasks/schema.sql`)

```sql
create table scheduled_tasks (
  id uuid primary key default gen_random_uuid(),
  name text not null unique,
  description text,
  enabled boolean default true,

  -- Trigger configuration
  trigger_type text not null check (trigger_type in ('cron', 'due_date', 'event', 'manual')),
  cron_expression text,
  due_date_source text,
  due_date_lead_days int default 1,
  event_source text,
  event_lead_hours int default 2,

  -- Data gathering
  gather_config jsonb not null default '{}',

  -- Execution
  task_type text not null check (task_type in (
    'llm_prompt', 'alert_digest', 'deck_builder', 'stale_loop_scan', 'trend_analysis'
  )),
  prompt_template text,
  deck_template_id text,
  output_format text default 'markdown' check (output_format in ('markdown', 'html', 'pptx', 'json')),

  -- Delivery
  delivery_channel text not null default 'email' check (delivery_channel in ('email', 'telegram', 'slack', 'file', 'mcp_response')),
  delivery_config jsonb default '{}',

  -- Metadata
  last_run_at timestamptz,
  last_run_status text,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create table task_run_log (
  id uuid primary key default gen_random_uuid(),
  task_id uuid references scheduled_tasks(id) on delete cascade,
  started_at timestamptz default now(),
  completed_at timestamptz,
  status text not null default 'running' check (status in ('running', 'success', 'error')),
  input_summary text,
  output_summary text,
  error_message text,
  delivery_status text
);

create index idx_task_runs_task_id on task_run_log(task_id);
create index idx_scheduled_tasks_trigger on scheduled_tasks(trigger_type) where enabled = true;
```

### MCP tools

Add to a new or existing Edge Function (follow existing `open-brain-mcp` pattern):

| Tool | Params | Description |
|------|--------|-------------|
| `create_scheduled_task` | `name`, `trigger_type`, `task_type`, `gather_config`, `delivery_channel`, + trigger-specific params | Register a new task. |
| `update_scheduled_task` | `id`, any mutable field | Update task config. |
| `list_scheduled_tasks` | `enabled_only?` (default true) | Show all tasks with last run status. |
| `run_task_now` | `id` or `name` | Manually trigger any task. |
| `task_run_history` | `task_id?`, `limit?` (default 10) | View recent run logs. |

### Runner logic (manual trigger only for this step)

When `run_task_now` is called:
1. Fetch the task definition
2. Execute gather phase using `gather_config` (query the brain tables)
3. Call the task type handler (stub for now — return "task_type X not yet implemented")
4. Log to `task_run_log` with timing and status
5. Return the result via `mcp_response` delivery channel
