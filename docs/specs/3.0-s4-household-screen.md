## Context for Claude Code

This is session 4 of 8. The CRUD pattern was established in S3 (Actions). This session replicates that pattern across three related but distinct data domains, all under one tabbed screen.

**Data scoping:** All three tables (`maintenance_tasks`, `household_items`, `vendors`) have `household_id` and are scoped by RLS: `household_id in (select household_id from household_members where user_id = auth.uid())`. Both Lee and Liv see the same data.

**Two MCP servers, three tables:**
- `home-maintenance-mcp` → `maintenance_tasks` table + `maintenance_log` table
- `household-knowledge-mcp` → `household_items` table + `vendors` table

**Reference the S3 pattern:** Look at `lib/queries/actions.ts` and `components/actions/` for the established patterns. Replicate the same structure: one query file per domain, domain-specific card/detail/form components, shared components for everything else.

## Maintenance tab

### List view

**Grouped by urgency using SectionHeaders:**

1. **Overdue** — `next_due < today` (red badge: "X days overdue")
2. **Due this week** — `next_due` within the next 7 days (amber badge: "X days")
3. **Upcoming** — `next_due` within the next 30 days (green badge: "X weeks")
4. **Later** — `next_due` beyond 30 days (blue badge: date)
5. **No schedule** — `frequency_days` is null and `next_due` is null (one-time tasks with no date)

Each group only renders if it has items.

### MaintenanceCard

```tsx
// Maps a maintenance_task to Card props
<Card
  title={task.name}
  subtitle={buildSubtitle(task)}            // e.g., "HVAC · Quarterly"
  badges={buildBadges(task)}
  rightContent={<Badge label={urgencyLabel} variant={urgencyVariant} />}
  onTap={() => openDetail(task)}
/>
```

**Subtitle logic:** Category + frequency in human-readable form:
- `frequency_days = 90` → "Quarterly"
- `frequency_days = 365` → "Annual"
- `frequency_days = 30` → "Monthly"
- `frequency_days = 14` → "Every 2 weeks"
- `frequency_days = null` → "One-time"
- General: `frequency_days` → "Every X days"

Include category if present: "HVAC · Quarterly"

**Priority badge:** If priority is `high` or `urgent`, show a red/amber badge.

### MaintenanceDetail

DetailPanel content when a task card is tapped.

**Sections:**
1. **Name** — full task name
2. **Category** — if set
3. **Priority** — badge
4. **Schedule** — frequency in human-readable form + next due date
5. **Last completed** — formatted date, or "Never" if null
6. **Notes** — if set
7. **Maintenance history** — list of past log entries (see MaintenanceHistory below)

**Header buttons:**
- Edit → MaintenanceForm in edit mode
- Log completion → LogMaintenanceDialog

**Footer (owner only):**
- Delete → ConfirmDialog

### MaintenanceForm

**Fields:**
- Name (text, required)
- Category (select: HVAC, Plumbing, Exterior, Appliance, Landscaping, Electrical, General — allow custom via "Other" option or free text)
- Priority (select: Low, Medium, High, Urgent, default Low)
- Frequency (number, optional, label: "Repeats every X days", placeholder: "e.g., 90 for quarterly")
- Next due (date, optional)
- Notes (textarea, optional)

**Helper text below frequency field:** "Common: 30 = monthly, 90 = quarterly, 180 = semi-annual, 365 = annual"

### LogMaintenanceDialog

Specialized modal for logging a task completion. This is different from editing — it creates a `maintenance_log` entry and updates the task's `last_completed` and `next_due`.

```tsx
type LogMaintenanceDialogProps = {
  isOpen: boolean
  taskName: string
  onLog: (entry: MaintenanceLogEntry) => Promise<void>
  onCancel: () => void
}

type MaintenanceLogEntry = {
  completed_at?: string       // defaults to now
  performed_by?: string       // 'self', vendor name
  cost?: number
  notes?: string              // work performed
  next_action?: string        // recommendations for next time
}
```

**Layout:**
- Title: "Log completion"
- Context: task name (read-only, muted)
- Performed by (text, optional, placeholder: "Self, vendor name, etc.")
- Cost (number, optional, placeholder: "$")
- Notes (textarea, optional, placeholder: "What work was performed?")
- Next action (text, optional, placeholder: "Recommendations for next time")
- Cancel + Log buttons

**On log:**
1. Insert into `maintenance_log`: `{ task_id, completed_at, performed_by, cost, notes, next_action }`
2. Update `maintenance_tasks`: set `last_completed = now()`, calculate `next_due = today + frequency_days` (if recurring)
3. Refresh the task list

**Note:** Check if the MCP `log_maintenance` tool handles step 2 automatically. If so, just insert the log entry via the same endpoint logic and let the backend handle `next_due` recalculation. Don't duplicate logic.

### MaintenanceHistory

Rendered inside MaintenanceDetail below the task info. Shows past completions.

```tsx
// Query: maintenance_log where task_id = this task, ordered by completed_at desc
```

**Each entry shows:**
- Date (formatted)
- Performed by (if set)
- Cost (if set, formatted as currency)
- Notes (if set, truncated to 2 lines)
- Next action (if set, muted italic)

If no history: "No maintenance logged yet."

Limit to last 5 entries with a "Show all" link that expands to full list.

## Items tab

### List view

**Grouped by category using SectionHeaders** (e.g., "Paint", "Appliance", "Measurement"). Categories derived from the data — no hardcoded list. Sorted alphabetically within each group.

**Search bar** at the top — filters items by name, category, and location (client-side filter on the loaded dataset, or Supabase `ilike` query).

### ItemCard

```tsx
<Card
  title={item.name}
  subtitle={buildSubtitle(item)}            // e.g., "Kitchen · Paint"
  onTap={() => openDetail(item)}
/>
```

**Subtitle:** Location + category, separated by " · ". Omit either if null.

### ItemDetail

**Sections:**
1. **Name** — item name
2. **Category** — if set
3. **Location** — if set
4. **Details** — render the `details` JSONB as key-value pairs (e.g., "Brand: Sherwin Williams", "Color: Sea Salt", "Finish: Eggshell"). This is flexible metadata, so render whatever keys exist.
5. **Notes** — if set
6. **Created** — formatted date

**Header buttons:** Edit → ItemForm
**Footer (owner only):** Delete → ConfirmDialog

### ItemForm

**Fields:**
- Name (text, required, placeholder: "e.g., Living room paint color")
- Category (select with common options: Paint, Appliance, Measurement, Document, Furniture, Fixture, Other — or free text)
- Location (text, optional, placeholder: "e.g., Kitchen, Master bedroom")
- Notes (textarea, optional)

**Details (the flexible metadata):** This is the tricky part. The `details` column is JSONB, meaning it can hold arbitrary key-value pairs. For v1, use a simple approach:

- Render existing details as a list of key-value rows (each with a text input for key and value)
- "Add field" button to add a new key-value row
- On save, collect all non-empty rows into a JSON object

This is a dynamic form section — the number of key-value rows varies per item. Keep the UI simple: each row is two text inputs side by side (key | value) with a remove button.

```
Details
┌──────────────┐ ┌──────────────┐  ✕
│ Brand        │ │ Sherwin W.   │
└──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐  ✕
│ Color        │ │ Sea Salt     │
└──────────────┘ └──────────────┘
         [+ Add field]
```

### List view

**Filter by service type:** Horizontally scrollable row of service type pills (derived from data, like the tag filter on Actions). "All" pill to clear filter.

Sorted alphabetically by vendor name.

### VendorCard

```tsx
<Card
  title={vendor.name}
  subtitle={vendor.service_type || ''}
  rightContent={vendor.rating ? <RatingDisplay rating={vendor.rating} /> : null}
  onTap={() => openDetail(vendor)}

```

**RatingDisplay:** Simple — show the rating as "★ 4.5" or render X filled stars out of 5. Keep it minimal — text is fine for v1.

### VendorDetail

**Sections:**
1. **Name** — vendor name
2. **Service type** — if set
3. **Rating** — star display
4. **Phone** — tappable link (`tel:` href) so it opens the phone dialer on mobile
5. **Email** — tappable link (`mailto:` href)
6. **Website** — tappable link (opens in browser)
7. **Last used** — formatted date, or "Never" if null
8. **Notes** — if set

**Phone, email, and website are the key UX elements here.** These need to be prominent and easily tappable — the main reason someone opens a vendor record is to call or email them. Make phone and email visually distinct (accent colored, large tap targets).

**Header buttons:** Edit → VendorForm
**Footer (owner only):** Delete → ConfirmDialog

### VendorForm

**Fields:**
- Name (text, required)
- Service type (text, placeholder: "e.g., Plumber, Electrician, Landscaper")
- Phone (text, optional — use `type="tel"` for mobile keyboard)
- Email (text, optional — use `type="email"`)
- Website (text, optional — use `type="url"`)
- Rating (number, optional, 1–5 — could be a simple select or a star picker, keep it simple)
- Last used (date, optional)
- Notes (textarea, optional)

## Data fetching pattern

Follow the S3 pattern. One query file per domain:

### lib/queries/maintenance.ts

```ts
// getTasks() — all maintenance_tasks, ordered by next_due (nulls last)
// getTask(id) — single task with its maintenance_log entries
// createTask(task) — insert into maintenance_tasks
// updateTask(id, updates) — update maintenance_tasks
// deleteTask(id) — delete from maintenance_tasks (owner only)
// logCompletion(taskId, entry) — insert into maintenance_log + update task
// getHistory(taskId) — maintenance_log entries for a task
```

### lib/queries/household-items.ts

```ts
// getItems() — all household_items, ordered by category then name
// getItem(id) — single item
// createItem(item) — insert (parse details JSON string to jsonb)
// updateItem(id, updates) — update
// deleteItem(id) — delete (owner only)
// searchItems(query) — search by name/category/location
```

### lib/queries/vendors.ts

```ts
// getVendors(serviceType?) — all vendors, optionally filtered
// getVendor(id) — single vendor
// createVendor(vendor) — insert
// updateVendor(id, updates) — update
// deleteVendor(id) — delete (owner only)
// getServiceTypes() — distinct service_type values for filter pills
```

## Delete behavior

Same as S3: owner only, ConfirmDialog, delete on confirm. The screen checks `isOwner` from AuthContext before passing `onDelete` to DetailPanel.

**Cascade warning for maintenance tasks:** Deleting a task should also delete its `maintenance_log` entries. Check if the database has `ON DELETE CASCADE` on the `task_id` foreign key. If not, delete log entries first, then the task. Surface this in the confirm dialog: "Delete this task and all its maintenance history? This can't be undone."

## Edge cases

1. **No maintenance tasks:** EmptyState: "No maintenance tasks. Tap + to add one."
2. **No household items:** EmptyState: "No household items recorded. Tap + to add one."
3. **No vendors:** EmptyState: "No service providers saved. Tap + to add one."
4. **Item with no details JSONB:** Omit the details section in ItemDetail. In ItemForm, show just the "Add field" button.
5. **Item with deeply nested details:** Only render top-level keys as key-value pairs. Don't try to handle nested objects in v1.
6. **Vendor with no contact info:** Show muted text "No phone, email, or website on file" instead of empty space.
7. **Maintenance task with very long history:** Show last 5 entries, "Show all" expands the rest.
8. **Category with only one item:** Still show the SectionHeader — don't collapse single-item groups.
